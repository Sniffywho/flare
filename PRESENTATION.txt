SLIDE 2 — PROBLEM STATEMENT
================================================================================

Communication tools today are fragmented:
  • WhatsApp-style apps focus only on private/group chats
  • Discord-style apps focus only on community servers with channels
  • Very few tools combine both models in a unified platform

PROBLEM:
  Students and small teams need a unified messaging platform that supports
  personal chats, group messaging, and structured community spaces — all
  in real time, without relying on external paid services.

SOLUTION:
  Build a self-hosted, full-featured chat application combining private
  messaging (WhatsApp-style) and community servers with channels
  (Discord-style), with real-time voice communication.


================================================================================
SLIDE 3 — PROJECT OVERVIEW
================================================================================

ChatterBox is a full-stack, real-time chat application that enables:

  ✦ Private (1-on-1) messaging
  ✦ Group chats with admin controls
  ✦ Discord-style community servers with text & voice channels
  ✦ Real-time voice calls (peer-to-peer via WebRTC)
  ✦ Rich messaging (reactions, replies, attachments, read receipts)
  ✦ Secure authentication with JWT

The application is designed for teams, friend groups, and communities
that need a self-hosted alternative to commercial messaging platforms.

================================================================================

  1. AUTHENTICATION & USER MANAGEMENT
     • Secure registration and login with JWT
     • Access tokens (15 min) + refresh tokens (7 days) with auto-rotation
     • User profiles: avatar, bio, display name, online status

     • Invite codes (shareable, regenerable)

  4. RICH MESSAGING
     • Message types: text, image, video, audio, file
     • Emoji reactions (multiple emojis, multiple users per emoji)
     • Reply threading (quote-reply to any message)
     • Edit messages (with edit history stored)
     • Soft delete (content cleared, record retained)
     • Pin messages in chats/channels
     • Full-text search across chat history

  5. REAL-TIME FEATURES (Socket.IO)
     • Instant message delivery via WebSocket
     • Typing indicators ("User is typing…")
     • Online/offline/away/do-not-disturb presence
     • Read receipts (sent → delivered → seen)

  6. VOICE CALLS (WebRTC)
     • Peer-to-peer voice calls in voice channels
     • Mute microphone / deafen (mute all incoming audio)
     • Multiple users in a voice channel simultaneously
     • Socket.IO used as signaling server for ICE/SDP exchange

SLIDE 10 — SECURITY IMPLEMENTATION
================================================================================

  AUTHENTICATION SECURITY
  • Passwords hashed using bcryptjs with 12-round salt
  • Short-lived JWT access tokens (15 minutes)
  • Long-lived refresh tokens (7 days) stored hashed in DB
  • Token rotation on refresh — old tokens invalidated immediately
  • Single-device logout support

  TRANSPORT & API SECURITY
  • Helmet.js: HTTP security headers (CSP, X-Frame-Options, HSTS)
  • CORS: Restricted to configured frontend origin only
  • Rate Limiting:
      - Global API:       200 requests / IP / 15 minutes
      - Auth endpoints:    10 requests / IP / 15 minutes
      - Message sending:   60 messages / IP / minute
  • gzip compression on all responses

  INPUT SECURITY
  • express-validator: Full input validation on all POST/PATCH routes
  • XSS library: Sanitize all user-generated content
  • express-mongo-sanitize: Prevent MongoDB operator injection ($, .)
  • Soft deletes: Messages never physically deleted from DB

  AUTHORIZATION
  • Middleware-based protect() for all authenticated routes
  • Role-based access: restrictTo('admin', 'owner', ...) per route
  • Server-level roles: owner > admin > moderator > member
  • Channel permission overrides: per-role and per-user allow/deny

SLIDE 11 — VOICE CHAT IMPLEMENTATION (WebRTC)
================================================================================

  TECHNOLOGY: SimplePeer (WebRTC library) + Socket.IO (signaling)

  HOW IT WORKS:

  Step 1 — Join Voice Channel
    User clicks "Join Voice" → client emits voice:join to server
    Server adds user to voiceParticipants list
    Server broadcasts voice:user_joined to all users in channel
    New user receives voice:existing_users (list of peers already in channel)

  Step 2 — Peer Connection Setup
    For each existing user, new user creates a SimplePeer (as initiator)
    SimplePeer generates an SDP offer → emitted via voice:signal to peer
    Receiving peer creates SimplePeer (non-initiator), receives offer
    Generates SDP answer → sent back via voice:signal

  Step 3 — ICE Candidate Exchange
    Both peers exchange ICE candidates via voice:signal events
    Socket.IO server relays all signals between peers
    Once connected, audio streams flow directly peer-to-peer

  Step 4 — Audio Controls
    Mute: disable local MediaStream audio tracks (stops sending audio)
    Deafen: set all remote audio element volumes to 0 (stops receiving)
    Leave: emit voice:leave, destroy all peer connections, release mic

  RESULT: Low-latency, peer-to-peer audio with no media server needed

SLIDE 13 — CHALLENGES & SOLUTIONS
================================================================================

  CHALLENGE 1: Real-Time Message Delivery at Scale
  Problem  : Ensuring messages reach all participants instantly,
             even when a user has multiple browser tabs open.
  Solution : Socket.IO rooms (one room per chat/channel).
             Online user map tracks userId → Set of socketIds.
             Messages broadcast to room; all sockets of a user receive it.

  CHALLENGE 2: WebRTC Peer Connections for Voice
  Problem  : Browsers cannot connect directly without a signaling server.
             Managing multiple peer connections in a group voice channel.
  Solution : Socket.IO used as signaling relay for SDP and ICE candidates.
             SimplePeer library abstracts WebRTC complexity.
             Each user maintains N-1 peer connections (mesh topology).

  CHALLENGE 3: Token Security & Session Management
  Problem  : Short-lived tokens frustrate users; long-lived tokens are risky.
  Solution : Dual-token system — 15-min access token + 7-day refresh token.
             Refresh tokens stored hashed in DB and rotated on each use.
             Axios interceptor handles refresh transparently on 401.

  CHALLENGE 4: Preventing Spam & Abuse
  Problem  : Message flooding, brute-force login attempts.
  Solution : Layered rate limiting — strict on auth (10/15min),
             moderate on messages (60/min), global fallback (200/15min).

  CHALLENGE 5: Consistent Read Receipts
  Problem  : Tracking which users have seen which messages across devices.
  Solution : readBy[] array in Message model (userId + timestamp).
             Socket event message:read triggers DB update and broadcast.
             deliveredTo[] separately tracks delivery vs. seen status.
